## 1. 目标

本次重构的目标不是单纯改名字，而是把当前本地栈整理成更稳定、更容易理解、并且不会因为多模型同时启动而把 128GB 统一内存打爆的架构。

目标包括：

- 模型命名改成“看名字就知道模型本体和大小”
- 默认常驻服务缩小到最小必要集合
- 其他模型改成按模型名请求时自动拉起
- 视觉模型不再人为拆成文本版和视觉版两套名称
- 引入内存准入、驱逐、拒绝策略，避免按需加载时内存失控
- 继续保留 `4060` 控制台作为统一运维入口

## 2. 当前问题

当前架构虽然已经稳定运行，但存在以下问题：

1. 模型命名不统一
   - `qwen-extract`
   - `qwen-fast`
   - `qwen-fast-vl`
   - `qwen-deep`
   - `qwen-deep-vl`
   这些名字混杂了“用途命名”和“模型本体命名”，长期维护会越来越难理解。

2. profile 目前主要是“常驻组合”
   - `LiteLLM` 只会把请求转发到静态 `api_base`
   - `supervisor` 只知道 profile 和手动启动
   - 当前还不具备“请求到了再自动启动模型”的能力

3. 没有内存准入控制
   - 在 128GB 统一内存机器上，如果同时拉起多个 27B/26B/9B 模型，存在明显爆内存风险
   - 当前没有“预计内存不足就拒绝启动”的机制

4. 文本模型和视觉模型拆成两套名字
   - 使用时容易混乱
   - 配置层也重复

## 8. 内存准入策略

### 8.1 基本原则

按需加载前必须做内存准入判断，不能直接无条件启动模型。

采用三段式策略：

1. 准入判断
2. 必要驱逐
3. 仍不满足则拒绝启动

### 8.2 为什么不能只看“空闲内存”

本机是 macOS 统一内存架构，Apple 官方建议观察的核心指标不是单纯 free memory，而是 `Memory Pressure`。

原因是：

- `Memory Pressure` 同时综合了 free memory、swap rate、wired memory、file cached memory
- `Compressed` 持续升高，说明系统已经在用压缩来换取可用内存
- `Swap Used` 持续增长，说明真实 RAM 已经开始不够

因此，本方案不采用“只要看剩余多少 GB 就行”的单一判断，而采用：

1. 静态预算守门
2. 动态 macOS 内存压力守门

### 8.3 预算内存表（联网论证后修正版）

- `embed-m3`：`1.5GB`
- `qwen2.5-1.5b`：`3GB`
- `qwen3.5-9b`：`16GB`
- `omnicoder-9b`：`14GB`
- `qwen3.5-27b`：`40GB`
- `huihui-27b`：`35GB`
- `gemma-4-26b`：`36GB`

说明：

- 这些值不是单纯文件大小，而是“模型权重 + mmproj + KV cache + 运行缓冲”的保守预算
- 预算表已经考虑到：
  - 27B / 26B Q8 模型文件本身就在 `25GB ~ 27GB`
  - 9B Q8 模型文件本身在 `8.9GB`
  - 视觉模型还有额外 `~0.88GB ~ 1.1GB` 的 `mmproj`
- 本机已有实测锚点：
  - `huihui-27b` 文件约 `27GB`
  - 先前本机运行态 RSS 已观察到约 `31GB+`
- 因此预算必须高于文件大小，不能直接拿 GGUF 文件体积当运行内存

### 8.4 预算表成立的前提参数

以上预算不是无条件成立，而是建立在“不要把默认参数开得过激进”的前提下：

- 重型模型默认建议：
  - `parallel = 1`
  - `ctx-size` 以 `32768 ~ 65536` 为默认安全带
- 9B 模型默认建议：
  - `parallel = 1`
  - `ctx-size` 以 `32768 ~ 65536` 为主
- 小模型默认建议：
  - 不要为了常驻轻模型而默认给很高并发和超长上下文

原因：

- `llama.cpp` 官方示例已经明确表明 `-c` 与 `-np` 会一起决定总上下文分配
- 并行请求数会把总上下文和额外内存一并放大
- 这也是当前草案里 `131072` 或较高 `parallel` 不宜继续作为重型模型默认值的根本原因

结论：

- 预算表不是独立存在的
- 它和启动参数策略必须成套落地
- 如果未来确实需要长上下文，应单独拆出例如 `qwen3.5-27b-longctx` 这类显式模型名，而不是把高 `ctx-size` 强塞给所有请求

### 8.5 安全阈值（双重守门）

本机总内存：`137438953472 bytes`，约 `128GB`

不再采用“只有一个 96GB 数字”的单点策略，而采用三段式阈值：

- 自动准入优先阈值：`88GB`
- 条件准入上限：`96GB`
- 超过 `96GB`：直接拒绝启动

含义如下：

1. `预测预算总量 <= 88GB`
   - 允许自动准入
   - 这是默认安全区
   - 更适合“日常开发 + 本地模型”的长期使用

2. `88GB < 预测预算总量 <= 96GB`
   - 进入条件准入区
   - 只有在动态内存指标健康时才允许启动
   - 允许先驱逐其他按需模型后再判断

3. `预测预算总量 > 96GB`
   - 直接拒绝
   - 不进入硬顶内存尝试

为什么是这组数值：

- `88GB` 约等于物理内存的 `69%`
- `96GB` 约等于物理内存的 `75%`
- 这相当于给 macOS、自身桌面应用、浏览器、IDE、文件缓存、统一内存波动保留 `25% ~ 31%` 的安全空间
- 相比旧稿中的“16GB 硬保留”，这组阈值更符合 Apple 官方对 `Memory Pressure` 的实际判断逻辑

### 8.6 动态内存健康守门

在 `88GB ~ 96GB` 条件准入区间，不能只看静态预算，还必须检查：

1. `memory_pressure` 当前是否仍处于健康状态
   - 优先要求绿色
   - 如果已经持续黄 / 红，不允许再启动重型模型

2. `Swap Used` 是否已经明显增长
   - 少量 swap 不是绝对错误
   - 但如果 swap 已经持续偏高，或短时间内明显上升，说明系统已在靠磁盘顶内存

3. 当前是否已经存在明显压缩 / pageout 压力
   - 这意味着即使静态预算勉强可算过，系统真实体验也会快速恶化

第一版实现建议：

- `<= 88GB`：只做静态预算判断
- `88GB ~ 96GB`：静态预算 + `memory_pressure` + `swap` 联合判断
- `> 96GB`：不做额外尝试，直接拒绝

### 8.7 准入计算逻辑

请求按需模型时，supervisor 需要计算：

- 当前已运行模型预算总和
- 当前目标模型预算
- 预测启动后预算总量
- 当前动态内存状态

如果：

`当前预算 + 目标预算 <= 88GB`

则允许启动。

如果：

`88GB < 当前预算 + 目标预算 <= 96GB`

则进入条件准入：

- 先尝试驱逐
- 驱逐后重新计算
- 只有动态内存状态健康时才允许启动

如果：

`当前预算 + 目标预算 > 96GB`

则直接拒绝，不再继续尝试

### 8.8 驱逐策略

不可自动驱逐：

- `litellm`
- `embed-m3`
- `qwen2.5-1.5b`

可自动驱逐：

- `qwen3.5-9b`
- `qwen3.5-27b`
- `omnicoder-9b`
- `huihui-27b`
- `gemma-4-26b`

驱逐顺序建议：

1. 最久未使用的按需模型优先停掉
2. 重型模型优先按照互斥组和预算回收
3. 同类模型优先被新请求替换

### 8.9 重型模型并存规则

这里修正旧稿中“超大模型最多一个”的结论。

对于 128GB 统一内存机器，更稳妥也更不保守的规则应是：

- `>= 30GB` 预算级别的重型模型，不允许自动启动第 3 个
- 第 2 个重型模型是否允许并存，由总预算和动态内存健康共同决定
- 常态运维仍然建议只长期热载 1 个重型模型

也就是说：

- `qwen3.5-27b + huihui-27b` 在某些时刻理论上可并存
- 但如果再叠加 `gemma-4-26b`，或再叠加较重的 9B 模型后逼近 `96GB`，就必须拒绝

因此，最终规则不是“永远只能有一个重型模型”，而是：

- 默认优先单重型
- 条件允许双重型
- 禁止第三个重型模型自动准入

### 8.10 超内存时的处理方式

按需请求最终只允许 3 种结果：

1. 直接启动成功
2. 先驱逐其他按需模型，再启动成功
3. 拒绝启动

拒绝时应返回明确错误，例如：

- `模型启动被拒绝：预计超过安全内存预算，请先停止其他大模型`
- `模型启动被拒绝：当前内存压力或 swap 状态不健康，不建议继续加载`

## 9. 按需加载链路方案

### 9.1 当前为什么还不能按需启动

当前架构下：

- LiteLLM 只知道静态 `api_base`
- supervisor 只知道 profile 和手动控制
- 两者之间没有“请求到了，请启动模型”的协议

### 9.2 最终建议方案

新增 supervisor 内部接口：

`POST /ensure-service`

请求示例：

```json
{
  "service": "qwen3.5-27b",
  "timeout_seconds": 120
}
```

由 supervisor 负责：

- 判断服务是否已经健康
- 做内存准入检查
- 必要时驱逐其他按需模型
- 启动目标服务
- 轮询直到健康或超时
- 返回结构化结果

### 9.3 LiteLLM 侧改造

在 LiteLLM 的本地 patch 中加入调用前钩子：

1. 根据 `model_name` 找到对应 service
2. 调用 `4060/ensure-service`
3. 返回 `ready` 才继续转发请求
4. 否则直接把错误返回客户端

### 9.4 并发控制

必须实现：

- 同一模型同一时间只能有一个启动流程
- 重型模型启动流程全局最好串行

否则可能同时触发多个大模型启动，内存判断失真。

## 11. 失败场景与处理策略

### 11.1 模型启动很慢

处理：

- `ensure-service` 支持较长超时
- 4060 页面显示“正在启动中”
- 超时后给客户端明确错误

### 11.2 模型文件缺失或参数错误

处理：

- `start_llama_model.sh` 已负责文件检查
- supervisor 记录 `last_error`
- `ensure-service` 返回结构化失败原因

### 11.3 并发请求重复触发启动

处理：

- 启动锁
- 相同模型合并等待

### 11.4 supervisor 不可用

处理：

- LiteLLM 直接返回明确错误
- 不再继续尝试转发

### 11.5 启动后长期不使用

处理：

- 第一版可先不自动停
- 第二版可引入“闲置超时自动停止”

## 13. 最终结论

本次重构的最终方向确定为：

- 模型命名统一为真实模型实体
- 常驻服务固定缩小到 3 个
- 其他模型按请求按需加载
- qwen 与 gemma 的视觉能力统一并入模型本身
- profile 只负责预热组合
- 内存准入必须采用“静态预算 + 动态 memory pressure / swap”双重守门
- 安全阈值不再是旧稿里的单一 `96GB`，而是：
  - `<= 88GB` 自动准入优先区
  - `88GB ~ 96GB` 条件准入区
  - `> 96GB` 直接拒绝区
- 模型预算表以修订后的保守值为准：
  - `embed-m3 1.5GB`
  - `qwen2.5-1.5b 3GB`
  - `qwen3.5-9b 16GB`
  - `omnicoder-9b 14GB`
  - `qwen3.5-27b 40GB`
  - `huihui-27b 35GB`
  - `gemma-4-26b 36GB`
- 对于重型模型：
  - 常态建议只热载 1 个

  - 不允许自动启动第 3 个重型模型

## 14. 联网论证依据与本机锚点

### 14.1 Apple 官方依据

- Apple Activity Monitor 文档明确说明：
  - `Memory Pressure` 由 free memory、swap rate、wired memory、file cached memory 共同决定
  - `Compressed` 表示系统正在压缩内存以换取可用空间
  - `Swap Used` 表示系统已经开始把内存页换到磁盘
- Apple 另一页说明：
  - 绿色表示内存使用高效
  - 黄色表示可能最终需要更多内存
  - 红色表示当前需要更多内存
- 参考链接：
  - `https://support.apple.com/guide/activity-monitor/view-memory-usage-actmntr1004/mac`
  - `https://support.apple.com/guide/activity-monitor/if-memory-pressure-is-yellow-or-red-actmntr34865/mac`
  - `https://www.manpagez.com/man/1/memory_pressure/`

### 14.2 `llama.cpp` 官方依据

- `llama-server` 官方 README 示例：
  - `-c 16384 -np 4`
  - 明确对应“4 个并发请求，每个 4096 max context”
- 这说明：
  - `ctx-size` 不是纯粹的逻辑参数
  - `parallel` 会直接放大总上下文和内存占用
- 官方 README 同时列出多模态模型支持
  - 结合本机实际存在的 `mmproj` 文件，可以确定视觉模型还要额外承担投影权重内存
- 参考链接：
  - `https://github.com/ggml-org/llama.cpp`

### 14.3 Ollama 官方依据

- Ollama FAQ 明确说明：
  - 每个模型的并行请求会增加总上下文
  - 所需 RAM 会按 `NUM_PARALLEL * CONTEXT_LENGTH` 方向增长
  - 如果可用内存不足，请求会排队，旧模型空闲后会卸载以腾出空间
- 这与本方案的“准入 -> 驱逐 -> 再启动 -> 仍不足则拒绝”一致
- 参考链接：
  - `https://docs.ollama.com/faq`
